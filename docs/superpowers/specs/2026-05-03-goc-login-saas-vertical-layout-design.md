# GOC 01 登录页 SaaS 化 — 纵向分区布局设计（无 Tab）

**状态：** 已定方向（brainstorming）  
**日期：** 2026-05-03  
**约束摘要：** 外部客户与内部员工共用同一登录 URL（场景 B）；采用 **方案 2 纯纵向分区**，**不**与 Tab/分段切换（方案 1）混用。

---

## 1. 目标

- 对外：支撑 SaaS 获客路径——**注册入口** + 邮箱密码登录 + **登录按钮下方的找回密码入口** + **Google**（带推荐说明）。
- 对内：**仅飞书登录**；不设企业员工专用的邮箱/密码表单（与上方对外表单完全分离，避免混淆）。
- 同一 URL 下一屏内完成分流：**上方为账号体系主路径**，**下方仅飞书 + 旁注**，不靠分区大标题区分受众。
- **移动端**两段 **均完整展开**，不使用折叠。
- **不出现**「客户与合作伙伴」「企业员工登录」等 **区块级标题**；必要时仅用页面级欢迎语（如产品原有 `Welcome to GOC 01!`），分区依赖 **间距 + 分割线 + 按钮旁说明文案**。

---

## 2. 页面信息架构（纵向）

### 2.1 上半区 — 对外主路径（无分区标题）

| 元素 | 说明 |
|------|------|
| 邮箱、密码 | 主路径输入 |
| 验证码 | 置于登录链路中（策略可与后端风控对齐） |
| 主按钮 | `登录` |
| **找回密码**（必选） | 紧接在 **登录按钮下方**，**文本链**（勿与主按钮同级大块按钮），示例：`忘记密码？` / EN：`Forgot password?`；跳转至找回密码流程（邮箱验证 / 重置链接等由后端与合规定义） |
| 次级 | `创建账号` / `注册`（链接或按钮；布局上置于找回密码之后或与找回密码同一辅助行，以视觉层次：**登录 > 找回密码 ≈ 注册**） |
| Google | **Outlined 按钮**，主文案示例：`使用 Google 登录`（或与 EN 站一致的简短动词短语） |
| Google **旁注**（必选） | 说明推荐使用场景；**主文案**（可再润色）：`推荐 · 使用 Google 可快速完成注册并登录` |
| **不出现** | 上半区内 **飞书** 入口 |

**视觉：** 占据表单区 **主要纵向空间**，留白充足。

### 2.2 下半区 — 对内路径（无分区标题）

| 元素 | 说明 |
|------|------|
| **仅 OAuth** | **飞书** 按钮（含飞书品牌图标） |
| **旁注**（紧贴按钮旁或按钮下方小字，必选） | 标明受众；**主文案**（可再润色）：`企业飞书账号登录 · 限内部员工` |
| **不出现** | 本区域内 **邮箱、密码、验证码、邮箱登录按钮** |

**视觉：** 与上半区用 **全宽分割线 + 间距** 分隔；可用浅色底条或略收窄区域暗示「次要入口」，**不使用 Tab**。

---

## 3. OAuth 与账号策略

### 3.1 Google（仅上半区）

- 对外获客：**注册 + 登录** 均可经 Google 完成；旁注承担「降低试错成本」的作用。
- 旁注若字数受限，可缩短为：`推荐 · Google 一键注册/登录`，上线前由产品与法务确认措辞。

### 3.2 飞书（仅下半区）

- 仅供已通过飞书组织管理的内部账号；外部用户误点若后端拒绝，错误提示应说明「请使用上方邮箱或 Google」，必要时链至注册。

### 3.3 注册入口

- **注册** 仅出现在 **上半区**（邮箱注册流或 Google），下半区不提供「注册」以免语义混淆。

### 3.4 找回密码

- **仅服务于邮箱密码账号**：Google / 飞书用户若遗忘密码，应遵循各 IdP 自带流程；前端可在找回页文案中一句话提示。
- 入口位置：**登录按钮正下方**，避免用户扫视遗漏；小屏同样保留，不换行挤爆时可单独一行。

---

## 4. 文案与错误提示

- **禁用** 「其他方式登录」这类把 Google 与飞书 **扁平并列** 的旧标题；改为「上图账号登录 / Google → 下图飞书」的纵向叙事。
- 登录失败：错误提示贴近对应操作区（邮箱登录 vs Google vs 飞书）。
- 飞书 OAuth 失败或非内部租户：在 **飞书按钮附近** 展示说明，引导外部用户使用上半区。

---

## 5. 移动端

- 上半区、下半区 **顺序纵向排列**，**不折叠**；允许页面变长，依赖自然滚动。
- Google 旁注、飞书旁注在小屏可 **换行** 显示，保持可读性（字号≥12px，对比度合规）。

---

## 6. 验证码（体验优化方向，可选迭代）

- 算术验证码可增加 SaaS 摩擦；后续可与后端约定按需展示或替换隐形人机验证。
- 本设计不要求一期改动后端，仅在文档中记录优化方向。

---

## 7. 验收要点

- 页面 **无** 分区级业务标题；用户仍能读懂「先账号/Google，再飞书」。
- **登录按钮下方可见「忘记密码？」**，且可进入完整找回流程。
- 外部用户：主导路径为邮箱与 **带推荐说明的 Google**；飞书在下方且带「内部」旁注。
- 内部员工：**仅通过飞书**进入下半区流程，不会看到重复的邮箱密码表单。
- 移动端：两段始终可见，无需折叠。

---

## 8. 明确不包含

- Tab / Segmented control。
- 移动端对内区域折叠。
- 下半区邮箱密码登录。
- 区块标题「客户」「员工」等大字标题（页面级欢迎语除外，若产品保留）。
